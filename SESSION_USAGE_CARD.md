# SESSION_USAGE_CARD.md

## 目的

用最简单的分流规则，减少主会话被噪音污染。

## 1. 主会话（决策）

只放：
- 目标澄清
- 方案比较
- 关键判断
- 最终结论
- 对 Bruce 的汇报

不放：
- 大段日志
- 原始 shell 输出
- 长网页原文
- 连续 patch 细节
- 反复试错过程

一句话：主会话只保留高价值信息，不当施工现场。

---

## 2. 调研会话（Research）

适用场景：
- 搜资料
- 看网页
- 读长文档
- 做信息归纳

输出要求：
- 只回传结论、证据、链接
- 不回传整段原始材料
- 若形成稳定认知，再写入 `MEMORY.md` 或当日日志

一句话：调研会话负责挖料，不负责替主脑囤料。

---

## 3. 修复会话（Debug / Repair）

适用场景：
- 日志分析
- shell / 命令排查
- 配置修复
- patch / 权限 / 连通性问题

输出要求：
- 问题是什么
- 根因在哪一层
- 临时止血是什么
- 永久修复是什么
- 是否需要记入 `INCIDENT_LOG.md`

一句话：修复会话可以很脏，但收尾必须很干净。

---

## 什么时候必须分流

出现任一情况，就不要继续留在主会话：
- 要贴长日志
- 要连续跑多条命令
- 要读多个长网页/长文件
- 要反复试错
- 问题已经从“判断”变成“施工”

---

## 收尾规则

任何调研/修复会话结束时，至少回传这 3 件事：
1. 结论
2. 依据
3. 下一步建议

若结论具有长期价值：
- 写入 `memory/YYYY-MM-DD.md`
- 必要时再沉淀到 `MEMORY.md` 或 `WORKING_RULES.md`
