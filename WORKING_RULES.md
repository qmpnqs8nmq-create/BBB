# WORKING_RULES.md

用最少规则，提升系统稳定性；避免治理过度。

## 根基：系统思考

动手前过一遍这五个问题：
1. 这个问题只在这一个点上，还是多个地方都会受影响？
2. 改了这里，还有哪些模块/配置/链路会被连带影响？
3. 表面现象背后，根因在哪一层？
4. 我现在的修法，会不会让问题下次更容易发生？
5. 这个改动一周后还成立吗？升级后还成立吗？

## 快捷口令

| Bruce 说 | 我该做什么 |
|---|---|
| "五个维度" | 停下来，逐一过上面五个问题，重新分析再决定方案 |
| "规则3" | 停下来，按①→⑤重新走一遍 |
| "规则4" | 停下来，至少列2条路径，逐条评估，选全局最优 |
| "系统思考"/"全局最优" | 先过五个问题，再按规则3+4执行 |

## 规则 1：长期信息只认文件，不认聊天

确认的偏好、决策、跨session状态、修复经验，必须写入对应文件（MEMORY.md / HANDOFF.md / 当日日志 / INCIDENT_LOG.md）。

## 规则 2：主会话只做判断，脏活必须分流

- 要贴长日志、连续跑命令、读多个长文件、反复试错 → 开调研/修复session
- 调研/修复完成后只回传：结论、依据、下一步，不回传原始噪音

## 规则 3：先诊断再动手，先想清楚再改

**① 定位根因** — 运行层→状态层→上下文层→治理层，逐层查，不跳步，不把临时故障直接上升为永久制度

**② 改之前三问** — 影响什么？波及什么？改错了怎么回滚？

**③ 备份再动手** — 备份原文件或记录还原命令

**④ 3次熔断** — 同一问题修3次没解决，强制停下重新全局诊断

**⑤ 改完记录** — 关键变更记入 `INCIDENT_LOG.md`

## 规则 4：方案选择必须比较，不选最快选最优

1. 至少列出2条可行路径
2. 每条评估：解决了什么、引入了什么新复杂度
3. 识别局部最优陷阱：是不是只解决眼前但让整体更脆弱？
4. 对首选方案做一次反向挑战：最大风险是什么？什么情况下会失败？
5. 如果蓝军挑战产生了更优路径，回到第1步重新列路径（最多循环2次，超过则带当前最优方案+未解决风险交给 Bruce 决策）
6. 选全局最优，不选埋隐患的捷径

## 规则 5：fallback 只当保险丝，不当稳定性方案

- 主模型只保留1个最可靠fallback，尽量跨provider异构
- 认证问题与模型问题分开诊断

## 规则 6：重复性故障必须进病历

同类问题第二次出现 / 需要monkey-patch / 升级后可能复发 / 影响关键链路 → 写入 `INCIDENT_LOG.md`，格式见该文件模板。
