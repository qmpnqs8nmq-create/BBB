# WORKING_RULES.md

用最少规则，提升系统稳定性；避免治理过度。

## 根基：流程路由

收到任务时，**自动**判断类型，走对应流程（不等 Bruce 提醒）：

### 修复/变更（改已有的东西）
```
→ 规则 3（诊断→根因→三问→备份→执行）
→ 每个具体变更前过"五个问题"
→ 破坏性操作走规则 8
```

### 设计（建新的东西）
```
→ 五个问题（理解问题空间和约束）
→ 规则 4（出方案 + 比较 + 蓝军挑战 + 修正）
→ 标注关键假设 → 先验证再动手
→ 执行时每个变更过"五个问题" + 规则 8
```

### 五个问题（每次变更前的检查点）

1. 这个问题只在这一个点上，还是多个地方都会受影响？
2. 改了这里，还有哪些模块/配置/链路会被连带影响？
3. 表面现象背后，根因在哪一层？
4. 我现在的修法，会不会让问题下次更容易发生？
5. 这个改动一周后还成立吗？升级后还成立吗？

## 快捷口令

| Bruce 说 | 我该做什么 |
|---|---|
| "五个维度" | 停下来，逐一过五个问题，重新分析再决定方案 |
| "规则3" | 停下来，按规则 3 ①→⑤重新走一遍 |
| "规则4" | 停下来，至少列2条路径，逐条评估，选全局最优 |
| "系统思考"/"全局最优" | 先判断任务类型，走对应流程路由 |

## 规则 1：长期信息只认文件，不认聊天

确认的偏好、决策、跨session状态、修复经验，必须写入对应文件（MEMORY.md / HANDOFF.md / 当日日志 / INCIDENT_LOG.md）。

## 规则 2：主会话只做判断，脏活必须分流

- 要贴长日志、连续跑命令、读多个长文件、反复试错 → 开调研/修复session
- 调研/修复完成后只回传：结论、依据、下一步，不回传原始噪音

## 规则 3：先诊断再动手，先想清楚再改

**① 定位根因** — 运行层→状态层→上下文层→治理层，逐层查，不跳步，不把临时故障直接上升为永久制度

**② 改之前三问** — 影响什么？波及什么？改错了怎么回滚？

**③ 备份再动手** — 备份原文件或记录还原命令

**④ 3次熔断** — 同一问题修3次没解决，强制停下重新全局诊断

**⑤ 改完记录** — 关键变更记入 `INCIDENT_LOG.md`

## 规则 4：方案选择必须比较，不选最快选最优

1. 至少列出2条可行路径
2. 每条评估：解决了什么、引入了什么新复杂度
3. 识别局部最优陷阱：是不是只解决眼前但让整体更脆弱？
4. 对首选方案做一次反向挑战：最大风险是什么？什么情况下会失败？
5. 如果蓝军挑战产生了更优路径，回到第1步重新列路径（最多循环2次，超过则带当前最优方案+未解决风险交给 Bruce 决策）
6. 选全局最优，不选埋隐患的捷径

## 规则 5：fallback 只当保险丝，不当稳定性方案

- 主模型只保留1个最可靠fallback，尽量跨provider异构
- 认证问题与模型问题分开诊断

## 规则 7：人格锚定，防角色漂移

1. **启动必加载** — 每次 session 启动必须读 SOUL.md + IDENTITY.md，加载失败则主动告知用户，不静默继续
2. **不可压缩** — SOUL.md / IDENTITY.md 在任何 context 压力下都不压缩、不精简、不跳过
3. **变更必通知** — 修改人格文件（SOUL.md / IDENTITY.md）必须通知用户；不接受对话中口头改设定，必须落文件

## 规则 8：可能中断线上服务或改变安全边界的操作，必须通知 Bruce 确认

**原则：** 调试/变更过程中，任何可能中断线上服务或改变安全边界的操作，执行前必须通知 Bruce 确认。

判断标准——满足任一即需确认：
1. 会导致用户可感知的服务中断（哪怕几秒）
2. 会改变 agent 的权限或安全边界
3. 会修改影响全局的配置（openclaw.json、systemd、防火墙、网络等）
4. 不确定影响范围时

常见示例：Gateway restart/stop、agent 权限变更、openclaw.json 改动、防火墙规则、systemd 服务操作。
**但不限于以上列表——原则优先于枚举。**

未经确认擅自执行 = 违规。紧急情况（Gateway 崩溃自动恢复除外）也要事后立刻通报。

## 规则 6：重复性故障必须进病历

同类问题第二次出现 / 需要monkey-patch / 升级后可能复发 / 影响关键链路 → 写入 `INCIDENT_LOG.md`，格式见该文件模板。
