## 问题诊断

**它不是什么？** 不是一个**系统设计方法论**。它不会逼你：
- 先列多个方案再选
- 对选定方案做攻击性挑战
- 验证关键假设再动手
- 检查整体设计的每个环节

## 应该用什么方法论

对于**设计一个新系统/新方案**（不是修一个 bug），正确的流程应该是：

### 系统思考 → 方案 → 蓝军挑战 → 综合修正 → 最小执行

**① 系统思考**
- 这个需求的完整链路是什么？涉及哪些组件？
- 有哪些约束条件？（安全边界、权限模型、现有架构）
- 有哪些未知/假设需要验证？

**② 方案（至少两条路径）**
- 方案 A：描述 + 优缺点 + 需要的权限/配置变更
- 方案 B：描述 + 优缺点 + 需要的权限/配置变更
- 每条路径标注：关键假设是什么

**③ 蓝军挑战**
- 对首选方案逐环节攻击：
  - 安全：谁能被影响？最坏情况？
  - 可靠性：哪个环节最可能挂？假设成立吗？
  - 权限：是不是最小化了？每个工具/权限都必要吗？
  - 回滚：改错了怎么恢复？

**④ 综合修正**
- 蓝军发现的问题 → 修正方案或换路径
- 列出必须先验证的假设 → 设计验证步骤

**⑤ 最小执行**
- 先验证假设
- 验证通过 → 最小变更落地
- 破坏性操作 → 通知 Bruce 确认
