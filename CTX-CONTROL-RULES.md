# CTX-CONTROL-RULES.md

## 目标
控制所有 session 的上下文膨胀，避免 `100% context used`、`Summary unavailable`、长会话漂移与判断质量下降。

**适用范围：**
- `main`
- `chief`
- `chief-user`
- `strategy`
- `market`
- `execution`
- `coach`
- 后续新增 agent / workspace

这不是 chief 专属规则，而是 **全局 session 治理规则**。

---

# 一、总原则

1. 不把 ctx 问题当成模型问题优先解决，先从会话治理解决。
2. 不把 `/compact` 当主方案；优先用新会话 + handoff。
3. 不把所有基础文件都压扁；只压缩可压缩层，保留治理骨架。
4. 长内容优先写入文件，不反复堆在聊天里。
5. 一条会话尽量只跑一个主任务；不要把多个大主题塞进同一线程。

---

# 二、Session 生命周期：三层信号联动

## 核心原则：context 长度是第一判断标准，空闲时间是辅助条件

空闲不是切换理由，context 满了才是。短对话即使空闲很久也不该切。

## 第一层：Context 压力（第一判断标准）

| context% | 动作 | 是否通知 |
|---|---|---|
| 0%–60% | 正常推进，无论空闲多久都不切 | 不通知 |
| 接近 compaction | memoryFlush 自动存档 HANDOFF + 当日笔记 | 不通知（自动） |
| compaction 触发 | 系统自动压缩老内容 | 不通知（自动） |
| 60%–70% + 用户在线 | 写入 HANDOFF，不再开新大主题 | **简短通知**："内容较多，建议发 /new 开新对话，我已存好上下文。" |
| 60%–70% + 空闲 ≥ 3h | 下条消息进来时先更新 HANDOFF，再简短提示 /new | **简短通知**："建议发 /new 开新对话，我已存好上下文。" |
| 70%+ | 必须准备切新会话 | **主动通知**："context 较满，发 /new 继续，不丢上下文。" |

## 第二层：结构性判断（辅助信号）
- **信号：** 对话内容特征（非数字指标）
- **触发条件（任一）：** 已跨两个以上主题 / 已生成多个长文件 / 已多轮工具排障 / 已进入下一阶段
- **动作：** 写 HANDOFF → 建议 /new
- **通知：** 简短提示

## 第三层：系统兜底
- session.reset 设为 idle 24h，纯防极端情况（如 session 被遗忘）
- 正常情况下不会触发

---

# 三、通知 vs 自动的原则

**自动（用户无感）：**
- compaction + memoryFlush
- HANDOFF.md 增量更新
- 当日笔记写入
- session maintenance 清理

**通知（一句话提示）：**
- context 60%–70%："建议 /new"
- context 70%+："请 /new"
- 跨话题/跨阶段时

**绝不做的：**
- 不在用户不知情的情况下主动 /new（重置是用户的选择）
- 不连续多次催 /new（说一次就够）
- 不因为 context 高就降低回答质量

# 四、文件分层压缩规则

## 可强压缩文件
- `MEMORY.md`
- `USER.md`
- `TOOLS.md`

### 压缩原则
- 只保留稳定、长期、可复用信息
- 不放长过程
- 不放示例大段说明
- 不放临时任务细节

## 不可压缩文件（硬红线）
- `SOUL.md`
- `IDENTITY.md`

### 原则
- 任何 context 压力下都不压缩、不精简、不跳过
- 这是人格锚点，丢失等于角色漂移

## 谨慎压缩文件
- `AGENTS.md`
- `WORKFLOWS.md`
- 审计 / 治理类骨架文件

### 原则
- 可以去重、去废话
- 不能为了 ctx 牺牲边界、治理字段

---

# 五、内容外置规则

## 长内容应写入
- `memory/YYYY-MM-DD.md`
- 项目 brief / baseline / plan
- runbook
- audit / checklist
- config consolidation 文档
- handoff 文档

## 外置原则
- 先写文件，再继续长任务
- 能写文件就不要继续在聊天里叠长文
- 需要跨会话承接时，优先给 handoff，不优先继续硬扛
- 要贴长日志、连续跑命令、读多个长文件、反复试错 → 开调研/修复 session，完成后只回传结论、依据、下一步

---

# 六、handoff 规则

## 每次切新会话前，必须生成 handoff
### 长度目标
- 6–12 条 bullet
- 尽量控制在 300–500 字

### 必含内容
- 当前唯一任务
- 已完成事项
- 当前状态
- 未完成事项
- 下一步动作
- 必要的精确路径 / 标识符 / 命令

---

# 七、执行口径

## 最优先方案
- 文件沉淀
- 新会话切换
- handoff 摘要
- 单任务线程

## 次优补救方案
- `/compact`
- contributor 排查
- 模式级 workaround

---

# 八、一句话操作规则

- 先外置，再切会话，不硬扛长线程。
- 先治理结构，不迷信 compact。
- 先保住治理骨架，再优化上下文长度。
- 自动能做的不通知，该通知的只说一次。
- 这份规则对所有 session 生效，不是 chief 专属规则。

---

# 九、Skill 安装门控

安装任何新 skill 前，必须先完成 vetting，不过关不装。

**检查项：**
1. 读 SKILL.md，检查权限要求和 exec 命令是否有危险操作
2. 评估与现有 skill / cron / 系统配置的冲突
3. 判断是否有真实增量价值（现有工具组合能否替代）
4. 系统重量是成本——能不加就不加

**适用范围：** 所有 agent（main、chief 及所有 specialist）。无论从哪个入口发起安装，均需过此门控。
