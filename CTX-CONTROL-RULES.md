# CTX-CONTROL-RULES.md

## 目标
控制所有 session 的上下文膨胀，避免 `100% context used`、`Summary unavailable`、长会话漂移与判断质量下降。

**适用范围：**
- `main`
- `chief`
- `chief-user`
- `strategy`
- `market`
- `execution`
- `coach`
- 后续新增 agent / workspace

这不是 chief 专属规则，而是 **全局 session 治理规则**。

---

# 一、总原则

1. 不把 ctx 问题当成模型问题优先解决，先从会话治理解决。
2. 不把 `/compact` 当主方案；优先用新会话 + handoff。
3. 不把所有基础文件都压扁；只压缩可压缩层，保留治理骨架。
4. 长内容优先写入文件，不反复堆在聊天里。
5. 一条会话尽量只跑一个主任务；不要把多个大主题塞进同一线程。

---

# 二、文件分层压缩规则

## 可强压缩文件
- `MEMORY.md`
- `USER.md`
- `TOOLS.md`

### 压缩原则
- 只保留稳定、长期、可复用信息
- 不放长过程
- 不放示例大段说明
- 不放临时任务细节

## 谨慎压缩文件
- `SOUL.md`
- `AGENTS.md`
- `WORKFLOWS.md`
- 审计 / 治理类骨架文件

### 原则
- 可以去重、去废话
- 不能为了 ctx 牺牲人格、边界、治理字段

---

# 三、会话治理规则

## 1. 60%–70% ctx 时默认准备切新会话
这不是普通建议，而是默认执行的强建议；除非当前任务马上结束，否则应优先切新会话，而不是继续扩题。

### 执行口径
- 0%–60%：正常推进
- 60%–70%：默认准备切新会话，不再开启新大主题
- 70% 以上：视为高风险区；除非马上结束，否则应切新会话

### 优先顺序
1. 新开会话
2. 带短 handoff 摘要
3. 必要时再 compact

### 例外
若当前任务只差最后一步、明确马上结束、且不再展开新分析，可允许直接收尾，不机械切换。

## 2. 出现以下任一情况必须切新会话
- 已跨两个以上主题
- 已生成多个长文件
- 已经历多轮工具排障
- 已进入下一阶段
- UI 已提示高 context 使用率
- 已出现 `100% context used`

## 3. 聊天内保留什么
聊天里只保留：
- 一句话判断
- 关键依据
- 风险
- 建议动作
- 下一步唯一任务

不要在聊天里反复保留：
- 工具原始日志
- 全量长文
- 多轮大段解释
- 已经写入文件的完整内容

---

# 四、内容外置规则

## 长内容应写入
- `memory/YYYY-MM-DD.md`
- 项目 brief / baseline / plan
- runbook
- audit / checklist
- config consolidation 文档
- handoff 文档

## 外置原则
- 先写文件，再继续长任务
- 能写文件就不要继续在聊天里叠长文
- 需要跨会话承接时，优先给 handoff，不优先继续硬扛

---

# 五、handoff 规则

## 每次切新会话前，必须生成 handoff
### 长度目标
- 6–12 条 bullet
- 尽量控制在 300–500 字

### 必含内容
- 当前唯一任务
- 已完成事项
- 当前状态
- 未完成事项
- 下一步动作
- 必要的精确路径 / 标识符 / 命令

---

# 六、执行口径

## 最优先方案
- 文件沉淀
- 新会话切换
- handoff 摘要
- 单任务线程

## 次优补救方案
- `/compact`
- contributor 排查
- 模式级 workaround

---

# 七、一句话操作规则

- 先外置，再切会话，不硬扛长线程。
- 先治理结构，不迷信 compact。
- 先保住治理骨架，再优化上下文长度。
- 这份规则对所有 session 生效，不是 chief 专属规则。
